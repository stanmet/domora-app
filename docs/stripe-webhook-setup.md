# Вебхуки Stripe в Domora: настраиваются автоматически

Коротко: **вручную в интерфейсе Stripe ничего создавать не нужно.**
Приложение само создает нужные вебхуки через Stripe API.

## Что случилось и почему так

В Stripe теперь два формата вебхуков:

- **Snapshot (классический)**: событие приходит целиком, с полным объектом
  (платежом, возвратом, аккаунтом). Наш код рассчитан на этот формат.
- **Thin (новый, "тонкий")**: приходит короткое уведомление, данные надо
  дозапрашивать. Работает только с новыми событиями v2 (например, Accounts v2)
  и не поддерживает платежные события payment_intent.* и charge.* вообще.

Вебхук, созданный в дашборде, получился в формате thin, поэтому Stripe писал
"This event is not compatible with this destination" при попытке добавить
классические события. Формат существующего вебхука поменять нельзя.

## Как это решено

1. **Приложение само создает вебхуки.** При первом платеже или онбординге
   исполнителя оно проверяет через Stripe API, есть ли нужные endpoint'ы,
   и создает недостающие:
   - "Domora: платежные события платформы": холд подтвержден, холд отклонен,
     возврат, чарджбек;
   - "Domora: онбординг исполнителей": account.updated от connected-аккаунтов,
     по нему включаются выплаты исполнителю.

   Оба видны в Stripe: Developers, вкладка Webhooks, с пометкой "создан автоматически".
2. **Секрет подписи не нужен.** Обычно вебхук проверяют секретом, который надо
   вручную копировать в настройки хостинга. Вместо этого приложение проверяет
   подлинность строже: перечитывает событие напрямую из Stripe API по его id
   и работает только с тем, что вернул Stripe. Подделать такое нельзя.
3. **Старый thin-вебхук можно не трогать.** Его события приложение вежливо
   подтверждает и игнорирует, ошибок доставки в Stripe копиться не будет.
   Хотите порядок: можно удалить его в дашборде, но это не обязательно.

## Что нужно от вас

Ничего, кроме уже существующих настроек:

- `STRIPE_SECRET_KEY` задан в Vercel (уже сделано для платежей);
- `APP_URL` задан в Vercel и указывает на боевой адрес сайта,
  например `https://domora.vercel.app`. Без него приложение не знает,
  на какой адрес Stripe должен слать события.

## Как проверить, что работает

1. Сделайте тестовое бронирование на сайте с картой `4242 4242 4242 4242`.
2. Откройте [dashboard.stripe.com](https://dashboard.stripe.com), Developers,
   вкладка Webhooks: там появятся два endpoint'а с описанием "Domora ... (создан
   автоматически)".
3. У endpoint'а "платежные события платформы" в списке доставок будет событие
   `payment_intent.amount_capturable_updated` со статусом 200.

Локальная разработка не меняется: события на localhost доставляет команда
`stripe listen --forward-to localhost:3000/api/stripe/webhook` (Stripe CLI),
а ее секрет можно положить в `STRIPE_WEBHOOK_SECRET` в `.env`.

## Необязательная оптимизация

Переменные `STRIPE_WEBHOOK_SECRET` и `STRIPE_CONNECT_WEBHOOK_SECRET` остались
поддержаны: если задать в них секреты endpoint'ов (Stripe показывает секрет
только в момент создания endpoint'а, для автоматически созданных его можно
получить, пересоздав endpoint вручную), проверка события пройдет по подписи,
без дополнительного запроса к Stripe API. На корректность это не влияет,
только на скорость, поэтому можно спокойно оставить их пустыми.

## Почему мы не перешли на thin-формат

- Thin-событий для платежных объектов (payment_intent, charge) в общем доступе
  не существует: даже переделав код, мы не узнавали бы о возвратах и чарджбеках.
- Snapshot это официальный, рекомендованный и бессрочно поддерживаемый Stripe
  путь для платежей.
- Когда Stripe выведет thin-события для платежей в общий доступ, можно будет
  вернуться к вопросу, спешить некуда.
