# Настройка вебхуков Stripe для Domora

Инструкция для владельца. Технических знаний не требуется: делайте шаги по порядку.

## Что случилось и почему

В Stripe теперь два формата вебхуков:

- **Snapshot (классический)**: Stripe присылает событие целиком, вместе с полным объектом
  (платежом, возвратом, аккаунтом). Наш код рассчитан именно на этот формат.
- **Thin (новый, "тонкий")**: Stripe присылает только короткое уведомление, а данные надо
  дозапрашивать через API. Этот формат работает только с новыми событиями v2
  (например, Accounts v2) и **не поддерживает платежные события**
  payment_intent.* и charge.* вообще.

Ваш текущий вебхук создан в формате thin, поэтому Stripe пишет
"This event is not compatible with this destination" при попытке добавить
классические события. Это не лечится настройкой: thin-вебхук физически не может
принимать snapshot-события. Формат существующего вебхука поменять нельзя,
нужно создать новые.

**Решение**: удалить thin-вебхук и создать два классических (snapshot) вебхука.
Почему два: платежные события (холды, возвраты, чарджбеки) приходят от вашего
основного аккаунта, а событие "исполнитель прошел онбординг" (account.updated)
приходит от подключенных аккаунтов исполнителей (Connected accounts). В Stripe
это два разных вебхука, у каждого свой секрет подписи. Код приложения уже
умеет работать с обоими.

## Шаг 1. Удалить thin-вебхук

1. Откройте [dashboard.stripe.com](https://dashboard.stripe.com), убедитесь, что вы
   в нужном режиме (Test mode для теста, потом то же самое повторить в Live).
2. Слева внизу: **Developers**, затем вкладка **Webhooks**
   (в новом интерфейсе это раздел Workbench, вкладка Webhooks).
3. Найдите созданный ранее вебхук (destination), откройте его, в меню
   с тремя точками выберите **Delete** и подтвердите.

## Шаг 2. Создать вебхук для платежных событий

1. Там же нажмите **Add destination** (может называться **Create new destination**).
2. На шаге выбора событий:
   - Если есть выбор **Events from**, выберите **Your account**.
   - Если есть выбор **Payload style** или переключатель **Use thin events**
     (обычно в Advanced settings), выберите **Snapshot** / переключатель выключите.
   - Если предлагается выбрать версию API, выберите обычную версию
     (например 2025-02-24.acacia или ту, что по умолчанию), а не версию
     с пометкой preview.
3. В поиске по событиям найдите и отметьте ровно эти четыре:
   - `payment_intent.amount_capturable_updated` (холд по карте подтвержден)
   - `payment_intent.payment_failed` (банк отклонил холд)
   - `charge.refunded` (возврат проведен)
   - `charge.dispute.created` (чарджбек, замораживаем выплату)

   Проверка, что вы на верном пути: если в поиске эти события находятся,
   значит формат правильный (snapshot). Если их нет, значит где-то выбран
   thin или события v2, вернитесь к пункту 2.
4. Тип destination: **Webhook endpoint**. В поле **Endpoint URL** вставьте:

   ```
   https://ВАШ-ДОМЕН/api/stripe/webhook
   ```

   Например: `https://domora.vercel.app/api/stripe/webhook`
5. Создайте destination. Откройте его страницу и найдите **Signing secret**
   (начинается с `whsec_`), нажмите Reveal и скопируйте.
6. Это значение пойдет в переменную `STRIPE_WEBHOOK_SECRET` (шаг 4).

## Шаг 3. Создать вебхук для онбординга исполнителей

1. Снова **Add destination**.
2. На шаге выбора событий выберите **Events from: Connected accounts**
   (accounts that your platform manages). Payload style так же snapshot.
3. Отметьте одно событие: `account.updated`.
4. Endpoint URL тот же самый: `https://ВАШ-ДОМЕН/api/stripe/webhook`.
5. Создайте, скопируйте его **Signing secret** (это другой `whsec_`, не тот, что в шаге 2).
6. Это значение пойдет в переменную `STRIPE_CONNECT_WEBHOOK_SECRET`.

## Шаг 4. Прописать секреты в Vercel

1. Откройте [vercel.com](https://vercel.com), проект domora,
   **Settings, затем Environment Variables**.
2. Добавьте или обновите две переменные:
   - `STRIPE_WEBHOOK_SECRET` = секрет из шага 2
   - `STRIPE_CONNECT_WEBHOOK_SECRET` = секрет из шага 3
3. Сохраните и сделайте **Redeploy** (вкладка Deployments, три точки у последнего
   деплоя, Redeploy), иначе новые переменные не подхватятся.
4. Если запускаете локально, добавьте те же две строки в файл `.env`.

## Шаг 5. Проверить, что все работает

1. В Stripe откройте созданный в шаге 2 destination. На его странице есть
   список доставок (Event deliveries / Attempts).
2. Сделайте тестовое бронирование на сайте с тестовой картой `4242 4242 4242 4242`.
3. В списке доставок должно появиться событие
   `payment_intent.amount_capturable_updated` со статусом успех (200).
4. Если статус 400 и в ответе видно `snapshot_webhook_required`, значит
   destination все еще thin: удалите его и повторите шаг 2.
5. Если статус 400 и `bad_signature`, значит в Vercel вставлен не тот секрет
   или забыли Redeploy.

## Почему мы не стали переделывать код под thin-формат

- Thin-событий для платежных объектов (payment_intent, charge) в общем доступе
  не существует: даже переделав код, мы не смогли бы узнавать о возвратах
  и чарджбеках. Snapshot-вебхук это официальный и рекомендованный путь для платежей.
- Snapshot надежнее для денег: событие приходит с полным объектом и проверенной
  подписью, без дополнительных запросов к API, которые могут упасть.
- Когда Stripe выведет thin-события для платежей из preview в общий доступ,
  можно будет вернуться к этому вопросу, спешить некуда: старый формат
  Stripe поддерживает бессрочно.
